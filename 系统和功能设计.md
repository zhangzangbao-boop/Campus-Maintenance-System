# 理工管家 - 校园智能报修中心系统和功能设计

## 1. 文档目标

本文档用于指导“理工管家 - 校园智能报修中心”的后续完善与开发实施。设计依据包括：

- 《软件系统设计与开发实践》课程任务书。
- 《理工管家 - 校园智能报修中心需求分析文档》。
- 当前项目已实现的 Spring Boot + React 系统功能。

课程任务书要求系统具备完整 Web 业务闭环、前后端交互、数据库持久化、登录注册、权限区分、异常处理、模块化分层、美观界面和课程报告支撑材料。因此本系统后续目标不是盲目堆功能，而是围绕“师生报修服务闭环”做成一个逻辑完整、稳定可演示、具备上线形态的校园后勤服务平台。

## 2. 当前系统定位

系统名称：理工管家 - 校园智能报修中心

系统定位：面向校园师生、维修人员和后勤管理员的智能报修与维修协同平台，解决传统报修渠道分散、流程黑盒、进度不透明、服务质量难量化等问题。

核心价值：

- 为学生提供统一、清晰、可追踪的报修入口。
- 为维修人员提供任务接收、处理、反馈的工作台。
- 为管理员提供工单分配、进度监控、用户管理、评价管理和数据分析能力。
- 通过状态机、评价体系和数据看板形成“提交 - 派单 - 处理 - 评价 - 分析 - 优化”的业务闭环。

## 4. 角色设计

### 4.1 是否需要新增角色

当前三种角色是合理的，不建议为了复杂度强行新增角色。

现有角色已经覆盖报修业务闭环：

- 学生：问题发起方、进度追踪方、服务评价方。
- 维修人员：任务执行方、维修记录填写方、完工确认方。
- 管理员：流程调度方、系统管理方、质量监督方、数据分析方。

对于课程项目，三角色更利于实现稳定闭环，也更符合需求分析文档。新增角色会带来权限矩阵、页面入口、测试数据和业务流程复杂度上升，若实现不完整反而降低系统质量。

### 4.2 可选扩展角色

如后续时间充足，可增加“宿管/楼栋负责人”作为二期增强，但不作为一期必做。

宿管/楼栋负责人可承担：

- 对本楼栋报修单进行初审。
- 标记现场是否属实、是否紧急。
- 查看本楼栋报修统计。
- 参与重复报修确认。

该角色的价值是让系统更贴近真实校园管理，但会增加流程节点。一期建议先保持 STUDENT、STAFF、ADMIN 三角色，二期通过权限配置扩展为 BUILDING_MANAGER。

## 5. 核心业务闭环设计

### 5.1 工单状态机

当前后端状态枚举为：

| 状态 | 中文含义 | 说明 |
| --- | --- | --- |
| WAITING_ACCEPT | 待受理 | 学生提交报修后进入该状态 |
| IN_PROGRESS | 处理中 | 管理员已派单给维修人员 |
| RESOLVED | 已处理 | 维修人员完成维修 |
| WAITING_FEEDBACK | 待评价 | 等待学生确认和评价 |
| FEEDBACKED | 已评价 | 学生已提交评分和评论 |
| CLOSED | 已关闭 | 流程归档结束 |
| REJECTED | 已驳回 | 管理员驳回无效报修 |

建议标准状态流：

```text
学生提交
  -> WAITING_ACCEPT
管理员派单
  -> IN_PROGRESS
维修人员完成
  -> RESOLVED
学生确认完成
  -> WAITING_FEEDBACK
学生评价
  -> FEEDBACKED
系统/管理员归档
  -> CLOSED
```

异常状态流：

```text
WAITING_ACCEPT -> REJECTED
IN_PROGRESS -> REJECTED
WAITING_ACCEPT -> 学生取消/软删除
```

建议后续统一前后端语义：学生端看到“维修人员完成后待我评价”时，可以把 `RESOLVED` 和 `WAITING_FEEDBACK` 都展示为“待评价”，但后端状态仍保持精确。

### 5.2 闭环流程

1. 学生登录后提交报修，填写分类、位置、描述、优先级、图片。
2. 系统进行表单校验和图片校验，创建工单和初始状态日志。
3. 管理员在工单池查看待受理工单，可按分类、楼栋、优先级、创建时间筛选。
4. 管理员选择维修人员派单，系统记录分配时间、预计完成时间和状态日志。
5. 维修人员查看“我的任务”，到场处理后填写维修备注、处理过程和图片。
6. 维修人员点击完成，工单进入已处理/待评价阶段。
7. 学生查看进度，确认问题解决后提交评分和评论。
8. 管理员查看数据看板，分析高发地点、高发类型、维修效率和满意度。
9. 系统通过统计结果反向优化维修资源配置和校园设施巡检计划。

## 7. 建议新增功能设计

### 7.1 学生端增强

#### 7.1.1 报修前智能引导

功能目标：降低学生填写成本，减少分类错误和重复报修。

建议能力：

- 根据问题描述自动推荐报修分类。
- 输入位置时给出常用楼栋/宿舍/教学楼提示。
- 上传图片后提醒图片数量、大小、格式是否合规。
- 提交前显示“相似报修提醒”，例如同一楼栋同类问题已有未完成工单。

落地方式：

- 一期：前端本地规则 + 后端按关键词查重。
- 二期：接入大模型进行分类推荐和问题摘要。
- 三期：保留本地相似工单规则检索，重点扩展 DeepSeek 在填单、摘要、报告和知识库生成中的使用范围。

#### 7.1.2 我的报修时间线

每个工单详情页增加时间线：

- 提交时间。
- 管理员受理/驳回时间。
- 派单时间。
- 维修人员开始处理时间。
- 维修完成时间。
- 学生评价时间。
- 管理员关闭时间。

该功能能明显体现“流程透明化”，也能增强教师验收时的观感。

#### 7.1.3 催单与补充说明

学生可在合理条件下操作：

- 工单超时未处理时发起催单。
- 在待受理/处理中阶段补充说明或追加图片。
- 查看管理员或维修人员的回复。

后端可新增表：

```text
repair_order_comment
- id
- repair_order_id
- user_number
- content
- image_url
- comment_type: SUPPLEMENT / REPLY / URGE
- created_at
```

#### 7.1.4 满意度反馈细化

当前评分和文字评价已经可用，可增强为：

- 维修速度评分。
- 维修质量评分。
- 服务态度评分。
- 是否彻底解决问题。
- 是否愿意匿名展示评价。

这样统计看板可展示更细粒度的服务质量。

### 7.2 维修人员端增强

#### 7.2.1 任务工作台优化

为维修人员提供更接近真实工作的任务视图：

- 今日待处理。
- 即将超时。
- 高优先级。
- 已完成待评价。
- 本月完成量和平均评分。

#### 7.2.2 维修过程记录

维修人员处理任务时增加：

- 到场确认。
- 维修前照片。
- 维修后照片。
- 维修耗材记录。
- 处理备注模板。
- 无法处理时申请转派或升级。

建议新增字段/表：

```text
repair_process_record
- id
- repair_order_id
- staff_id
- action_type: ARRIVED / MATERIAL_USED / FINISHED / TRANSFER_REQUEST
- content
- image_url
- created_at
```

#### 7.2.3 转派与协同

当维修人员发现工单不属于自己专业范围，可申请转派：

- 维修人员提交转派原因。
- 管理员审核并重新分配。
- 状态日志记录转派过程。

该功能让系统更贴近真实后勤管理。

### 7.3 管理员端增强

#### 7.3.1 智能派单

管理员派单时系统给出推荐维修人员：

推荐依据：

- 维修人员专业方向/擅长分类。
- 当前未完成任务数量。
- 历史平均处理时长。
- 历史评分。
- 楼栋/区域距离。

一期实现：

- 基于 SQL 统计的规则评分。

二期实现：

- 引入简单权重模型：

```text
score = 0.35 * 空闲度 + 0.25 * 分类匹配 + 0.2 * 平均评分 + 0.2 * 历史处理速度
```

#### 7.3.2 SLA 超时管理

| 优先级 | 响应时限 | 完成时限 |
| --- | --- | --- |
| high | 2 小时内受理 | 24 小时内完成 |
| medium | 8 小时内受理 | 3 天内完成 |
| low | 24 小时内受理 | 7 天内完成 |

管理员看板增加：

- 超时未受理工单。
- 超时未完成工单。
- 即将超时工单。
- 超时率趋势。

#### 7.3.3 热点问题分析

数据看板除了现有分类、地点、月度、评分之外，建议增加：

- 高频楼栋排行。
- 近 7 天异常增长分类。
- 重复报修地点。
- 各维修人员任务负载。
- 各分类平均处理时长。
- 评价关键词云。

#### 7.3.4 系统配置中心

管理员可配置：

- 报修分类。
- 楼栋/区域字典。
- 优先级规则。
- SLA 时限。
- 公告内容。
- 图片上传限制。

这能体现系统可维护性，而不是写死在代码里。

## 8. 特色功能设计

### 8.1 大模型智能报修助手

目标：让系统从普通 CRUD 管理系统升级为“智能校园服务平台”。

可实现功能：

- 学生输入自然语言，AI 自动提取分类、位置、问题摘要和紧急程度。
- AI 根据描述生成规范化报修标题。
- AI 给出临时安全提醒，例如“疑似电路问题，请勿自行拆卸插座”。
- 管理员查看工单时，AI 自动摘要问题要点。
- 维修人员填写处理记录时，AI 辅助生成标准维修报告。

示例交互：

```text
学生输入：宿舍 3 楼卫生间水一直漏，地面很滑。
AI 输出：
分类：水电维修
位置：宿舍 3 楼卫生间
优先级：high
标题：宿舍卫生间水管漏水
安全提醒：请暂时避开积水区域，防止滑倒或触电风险。
```

实现建议：

- 后端新增 `AiAssistantService`。
- 前端在报修表单中增加“智能填写”按钮。
- 大模型接口使用可配置 Provider，避免写死供应商。
- 如果课程演示环境无法联网，可准备 Mock 模式。

配置建议：

```yaml
ai:
  enabled: false
  provider: openai-compatible
  base-url:
  api-key:
  model:
```

### 8.2 本地相似工单检索

目标：识别重复报修，辅助管理员判断是否合并处理。

当前选择：删除独立语义检索数据库依赖。课程项目优先保证本地部署简单、演示稳定、功能闭环完整；相似工单继续由后端规则检索实现。

可实现功能：
- 学生提交前提醒“附近已有类似报修”。
- 管理员派单时查看相似历史工单。
- 维修人员查看历史解决方案。
- 统计“重复报修高发区域”。

数据流程：

```text
工单描述 + 位置 + 分类
  -> 关键词提取、分类匹配、地点匹配
  -> 历史工单打分排序
  -> 返回相似工单和历史处理结果
```

技术路线：

- 后端 `AiAssistantService.findSimilarTickets` 基于分类、位置、关键词交集、状态进行综合打分。
- 无需额外服务，MySQL 初始化后即可运行。
- DeepSeek 主要用于生成结构化报修分析和知识库内容，不承担历史工单检索。

### 8.3 知识库与维修方案推荐

把历史维修记录沉淀成知识库：

- 常见问题。
- 处理步骤。
- 所需工具/耗材。
- 安全注意事项。
- 预计处理时间。

管理员维护知识库，维修人员处理工单时自动推荐相关条目。

DeepSeek 增强：

- 管理员输入分类和故障现象后，AI 自动生成知识库草稿。
- 草稿包含标题、症状关键词、处理步骤、安全注意事项和预计处理时间。
- 管理员确认后保存，避免 AI 内容未经审核直接进入正式知识库。

建议新增表：

```text
repair_knowledge_base
- id
- category_key
- title
- symptom_keywords
- solution_steps
- safety_notes
- enabled
- created_at
- updated_at
```

### 8.4 校园设施健康指数

基于报修数据生成“设施健康指数”：

- 楼栋健康分。
- 分类故障风险。
- 重复报修率。
- 平均处理时长。
- 满意度。

展示形式：

- 管理员大屏。
- 校园地图/楼栋热力图。
- 重点风险区域提醒。

这是一个很适合答辩展示的特色点，因为它从单个工单上升到校园运维分析。

## 9. 后端系统设计

### 9.1 架构分层

后端采用标准 MVC + 分层架构：

```text
Controller 层
  接收请求、参数校验、权限注解、返回 DTO

Service 层
  业务规则、事务边界、状态机、异常抛出

Repository 层
  JPA 数据访问、统计 SQL、分页查询

Entity 层
  数据表映射、实体关系、枚举状态

DTO/Request/Response 层
  前后端数据契约，避免直接暴露实体

Config 层
  安全配置、JWT 过滤器、跨域、静态资源、定时任务

Exception 层
  统一异常响应、业务异常、参数校验错误
```

### 9.2 认证与权限

当前设计：

- 登录成功后生成 JWT。
- 前端存储 token，后续请求带 `Authorization: Bearer <token>`。
- 后端 JWT 过滤器解析 token 并注入 Spring Security 上下文。
- 管理员接口使用 `@PreAuthorize("hasRole('ADMIN')")`。

建议增强：

- JWT 密钥从配置文件读取，不硬编码。
- 增加 token 过期提示和前端自动登出。
- 高风险操作二次确认，例如恢复备份、删除用户、删除评价。
- 后端补全学生、维修人员接口的角色校验，避免只依赖前端页面入口。
- 增加接口级权限矩阵文档。

### 9.3 参数校验

当前已有 Jakarta Validation。建议补强：

- 登录账号不能为空、长度限制。
- 密码长度和复杂度。
- 手机号格式。
- 报修位置长度。
- 描述最少 10 字，最多 1000 字。
- 图片数量 0-3 张，单张大小限制。
- 评分必须 1-5。
- 驳回必须填写原因。
- 完成工单必须填写维修备注。

前端和后端都要校验，前端提升体验，后端保证安全。

### 9.4 异常响应格式

建议统一 API 响应：

```json
{
  "timestamp": "2026-06-27T14:00:00",
  "status": 400,
  "error": "Bad Request",
  "message": "报修地点不能为空",
  "path": "/api/repair-orders"
}
```

对于正常业务响应，可统一：

```json
{
  "code": 200,
  "message": "success",
  "data": {}
}
```

当前项目部分接口已接近该模式，后续建议整理一致。

### 9.5 并发与事务

已有能力：

- JPA `@Version` 乐观锁。
- 关键工单操作中使用悲观锁查询。
- Service 层 `@Transactional`。

建议保留并强化：

- 派单时防止同一工单被重复分配。
- 评价时防止重复评价。
- 状态变更必须通过状态机校验。
- 数据备份恢复不与普通业务操作并发执行。

## 12. 前端设计升级方案

### 12.1 总体视觉定位

前端应从“课程项目页面”升级为“成熟校园服务平台”：

- 风格：清爽、现代、可信赖、校园科技感。
- 颜色：以理工蓝/青绿为主，辅以状态色。
- 信息密度：管理员和维修工页面偏工作台风格，学生页面偏简洁引导。
- 布局：侧边栏 + 顶部用户区 + 内容工作区。
- 动效：轻量过渡、卡片 hover、状态流动画、图表动态加载。

### 12.2 登录页升级

建议：

- 左侧展示品牌：“理工管家 - 校园智能报修中心”。
- 右侧登录卡片，角色自动识别。
- 增加测试账号快捷填充，仅开发环境显示。

### 12.3 学生端界面

重点做“少步骤、强引导”：

- 待评价工单突出显示。

### 12.4 维修工端界面

重点做“任务工作台”：

## 13. 安全与稳定性设计

### 13.1 安全策略

- 密码使用 BCrypt 存储。
- JWT 鉴权，后端不使用 Session。
- 管理员接口必须强制角色校验。
- 文件上传限制格式、大小和路径。
- 防止越权访问：学生只能查看自己的工单，维修工只能查看自己的任务。
- 备份恢复、删除评价、禁用用户等操作写入审计日志。

### 13.2 稳定性策略

- 全局异常处理，避免后端直接暴露堆栈。
- 所有关键业务操作使用事务。
- 派单、评价等并发敏感操作加锁。
- 数据库添加索引，提升列表和统计查询性能。
- 备份恢复前强制创建最新备份。
- 初始化脚本保证演示环境可快速恢复。

## 19. 结论

当前三角色设计已经足够支撑课程要求，不建议一期强行新增角色。后续重点应放在以下方向：

- 把现有功能做稳：权限、校验、状态机、异常处理、数据一致性。
- 把业务闭环做完整：通知、时间线、维修过程、评价、统计分析。
- 把界面做成熟：工作台式布局、统一视觉、动态图表、清晰交互。
- 把特色做亮：AI 智能报修、相似工单检索、知识库、设施健康指数。

按照本文档推进后，系统不仅能满足课程任务书对小型 Web 系统的要求，也能在验收时体现工程完整度、业务闭环能力和一定的创新性。
